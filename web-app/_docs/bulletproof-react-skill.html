What this is

For the reviewer who landed here cold.

The team is migrating web-app/ toward Bulletproof React, with adjustments. Hunter Coleman scaffolded the canonical example at web-app/src/features/performance/met-station/ and wrote web-app/_docs/_README_FEATURES.md describing the deviations.

Cursor reads .cursor/rules/*.mdc automatically. Claude Code reads .claude/skills/<name>/SKILL.md. Both tools now point at the same source of truth — the SKILL.md is canonical; the .mdc mirrors it. The conformance script is comment-tagged accordingly.

Deviations from upstream Bulletproof React

All explicit; nothing accidental. See _docs/_README_FEATURES.md for Hunter's framing.

Aspect Upstream Proximal variant
Feature location features/<feature>/ flat features/<group>/<feature>/, grouped by product domain
Allowed subfolders api, assets, components, hooks, stores, types, utils Closed list of seven: components, hooks, queries, routes, types, utils, views
Barrel files "Avoid" Required index.ts per feature exporting only the route — boundary enforcement chosen over tree-shaking
Query/data hooks Inside the feature's api/ Mostly shared in @/api/v1/... and @/hooks/api; feature queries/ is rare
Routes src/app/routes/ Inside the feature (routes/<Feature>Route.tsx); paths still in App.tsx
File naming Everything kebab-case PascalCase .tsx for component-exporting files; kebab-case .ts for everything else
Migration affordance n/a Legacy page.tsx becomes a one-line shim importing the feature route — router stays untouched

The seven allowed subfolders

Closed list. Adding any other folder is an anti-pattern (and the conformance script will fail).

Folder Purpose
components Presentational/composite UI used by this feature
hooks React hooks: state, wiring, context consumers
queries Feature-local data hooks — used sparingly; most queries live in shared @/api/v1/...
routes Route-level entry components (<Feature>Route.tsx)
types TypeScript types for this feature
utils Pure helpers, builders, normalizers
views Larger screens or tab panels composed from components

File naming

Mixed by export type. Mechanical mapping in both directions.

PascalCase .tsx

Files whose primary export is a React component. Symbol = filename. 

routes/MetStationRoute.tsx
  export function MetStationRoute()

views/DayView.tsx
  export function DayView()

components/Tabs.tsx
  export function Tabs()

kebab-case .ts

Everything else. Function name → kebab-name is mechanical. 

hooks/use-met-station-context.ts
  export function useMetStationContext()

utils/build-time-search-params.ts
  export function buildTimeSearchParams()

types/met-station.ts
  export type MetStationContext, MetStationTab
Folders are always kebab-case. met-station never MetStation or met_station. The conformance script enforces this.

Context + view-model pattern

Hunter's signature pattern. Two layers cleanly separate "what this feature needs" from "what this view needs".

Layer File Returns
Context hook hooks/use-<feature>-context.ts { ids, entities, isLoading, error }
View-model hook hooks/use-<feature>-<view>-view-model.ts The shape that one view renders 
function MetStationRoute() {
  const { projectId } = useParams()
  const { activeTab, setActiveTab } = useMetStationTab()
  const context = useMetStationContext({ projectId })

  if (context.isLoading) return <PageLoader />
  if (context.error !== null) return <PageError text="..." />
  if (context.project == null) return <PageError text="Project not found" />

  return (
    <Tabs value={activeTab} onChange={setActiveTab}>
      {activeTab === 'realtime'    && <RealTimeView context={context} />}
      {activeTab === 'current-day' && <DayView      context={context} />}
      {activeTab === 'long-term'   && <LongTermView context={context} />}
    </Tabs>
  )
}

function DayView({ context }: { context: MetStationContext }) {
  const data = useMetStationDayViewModel({ context, start, end })
  return <Layout>...</Layout>
}
Why two layers? Loading and errors are checked once at the route. Each view receives a resolved context and adds only its per-view shaping. Adding a new view = one VM hook + one component + one tab branch.

Data layer — the biggest deviation

Most query hooks live outside the feature.

Most queries and data hooks are not co-located with their associated features. Our API is generalized enough that most endpoints touch multiple pages, so those hooks usually belong in shared API or query-layer locations anyway. Use a feature's queries folder only for hyper-specific hooks that provide truly individualized endpoint access, and do so sparingly.

— web-app/_docs/_README_FEATURES.md

In practice: shared hooks live in @/api/v1/... and @/hooks/api (which includes useCustomQuery). Both met-station and pv-inverter have no queries/ folder — by design.

App.tsx didn't change. The shim is the whole reason — the router still imports the same path; the implementation moved underneath.

  1. Pick destination. src/features/<group>/<feature>/.
  2. Scaffold only what you need from the seven allowed subfolders.
  3. Extract context hook (hooks/use-<feature>-context.ts) aggregating shared queries.
  4. Split per view: one VM hook + one <View> per tab.
  5. Relocate the rest: components, helpers, types into matching subfolders.
  6. Wire the route: routes/<Feature>Route.tsx.
  7. Public API: add index.ts, export only the route.
  8. Shim the legacy page with the 5-line import + render.
  9. Verify: pnpm check (now includes conformance) + manual smoke-test each tab.

Anti-rationalization table

Borrowed from Addy Osmani's pattern. Pre-written rebuttals stop step-skipping.

Excuse Rebuttal
"I'll inline the fetcher just for now." The fetcher belongs in @/api/v1/... or @/hooks/api. Inlining now means another refactor later. Do it once.
"The page imports from another feature — I'll keep it." No cross-feature imports. Lift the shared piece up first, then migrate.
"Adding index.ts is overhead." The barrel is the only public boundary. Without it, callers reach into internals and the boundary erodes.
"I'll skip the shim and update App.tsx." The shim limits the blast radius. Router edits change every reviewer's mental map of routing; shim edits don't.
"I'll use a side-effect import for the cross-feature dependency." Any cross-feature import is a no-go (absolute, relative, side-effect). Lift the shared piece to src/.
"I'll gate the whole view on isLoading." Per-card <Skeleton> keeps the layout stable. Gate the route, not the view.

Rule What it does

no-deep-feature-import From outside features, no @/features/<g>/<f>/<sub>/... imports — only the barrel

Tool-agnostic. The script reads the codebase, not the agent that wrote it. Whether changes come from Cursor, Claude Code, or a human, conformance is measured the same way. This independence keeps both Cursor (.mdc) and Claude Code (SKILL.md) rule files honest — neither needs its own enforcement layer.
